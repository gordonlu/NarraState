# NarraState 架构

NarraState（谜局AI）是状态驱动的互动叙事系统。核心原则是：

> Rust 决定发生什么，模型决定角色如何表达。

## 模块边界

```text
web
  ↓
narrastate-server
  ├── narrastate-case
  ├── narrastate-provider
  ├── narrastate-storage
  └── narrastate-runtime
          ↓
    narrastate-core
```

- `narrastate-core`：领域类型、不变量、案件校验和披露图；不依赖 Web、数据库或模型 SDK。
- `narrastate-runtime`：确定性证据评估、状态转换、对话计划以及 provider/repository ports。
- `narrastate-case`：案件草案规范化、变体编译、校验、模拟、内容哈希和案件包。
- `narrastate-provider`：OpenAI-compatible 文本与图片 Provider；输出不是权威状态。
- `narrastate-storage`：SQLite、冻结实例、会话、事件、快照、幂等结果和生成任务。
- `narrastate-server`：Axum API、SSE、CLI、DTO 脱敏和依赖组合。
- `web`：Vue 客户端；只展示公开状态，不计算压力、披露资格或结案结果。

## 案件生命周期

```text
作者文件或 AI 草案
        ↓
规范化与确定性编译
        ↓
结构和语义校验
        ↓
每个真相变体独立模拟
        ↓
安装版本化案件包
        ↓
按模式与 seed 选择变体
        ↓
冻结完整 CompiledCase 实例
        ↓
创建 Session
```

AI 生成结果必须先进入草案模型。未通过编译、校验和模拟的内容不能安装或游玩。会话始终读取数据库中的冻结实例，因此案件文件更新不会改变已有存档。

## 真相与状态

一个案件模板可以包含多个已经验证的真相变体。新游戏可以选择默认、随机或指定变体；固定 seed 会稳定复现选择。实例创建后，责任人、世界事实、证据链和结局条件不再改变。

玩家行动只能影响角色运行时状态，例如压力、信任、防御策略、审讯阶段和已解锁披露。矛盾数量或模型文本不能直接触发认罪；完整认罪只能经过 `DisclosureGraph` 的确定性前置条件。

## 知识隔离

系统维护三层相互独立的信息：

- 世界真相：冻结案件实例中的完整事实；
- 角色知识：角色定义、信念、陈述和运行时披露状态；
- 玩家知识：已经发现的事实、证据和公开对话。

普通 API 只返回玩家视角，不暴露真相变体、责任人、隐藏事实或完整披露图。

## 单轮处理

```text
校验 revision 与玩家可见引用
        ↓
模型或 Mock 解释玩家文本
        ↓
Rust 评估证据并转换状态
        ↓
生成受约束的对话计划
        ↓
模型渲染或确定性模板降级
        ↓
原子提交状态、事件、对话和 revision
```

Provider 失败必须可观察，并使用明确降级策略。模型响应永远不能绕过 Rust 状态机。

## 持久化与恢复

SQLite 保存已安装案件索引、冻结案件实例、玩家会话、事件日志、快照、幂等 action 结果、生成任务和脱敏后的调用元数据。API Key 保存在服务端环境变量或权限受限的本地 env 文件中，不进入案件包、浏览器响应或调用日志。

## 非证据性视觉

生成图片只具有 `Decorative` 语义角色，用于封面、角色、地点、章节、转场和结局氛围。视觉资产不进入事实、证据、矛盾、披露或结案链路；删除全部生成图片后，案件编译和模拟结果必须保持不变。

## 必须保持的不变量

- 新增领域约束必须有失败测试；
- 所有 ID 引用在案件加载时进行语义校验，并返回字段路径；
- 数值变化通过领域方法和饱和运算完成；
- 正常玩家 DTO 必须经过真相脱敏；
- 测试和 CI 不依赖真实模型 API；
- 不静默吞掉错误或伪造成功结果。

更具体的案件格式、生成、校验、模拟和 API 约定见 `docs/` 下对应主题文档。
